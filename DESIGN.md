# 擦水晶 · 水晶磨き · Crystal Puzzle — 理念文档

> 一款轻松解压的 3D 擦拭小游戏。Godot 4.6 / gl_compatibility，一套代码同发 **iOS / Web(GitHub Pages) / Android**。
> 场景全部脚本生成(`main.gd`)，无 `.tscn`。三语发行——名字随设备语言:繁/简中「擦水晶」、日「水晶磨き」、英「Crystal Puzzle」(App Store 商店名「Crystal Puzzle: Wipe & Polish」，因商店名须全球唯一)。

---

## 一、核心理念

### 1. 一句话
你是小镇上的**解谜大师**。为查清近来的流言，你翻出箱底那盒尘封的**水晶拼图**，喷水擦去灰尘，让蒙尘的水晶重现通透光泽——随着灰尘褪去、光芒透出，真相也逐渐浮现。

### 2. 情绪目标:解压
- 无失败、无倒计时、无强目标。玩家自由地擦。
- 反馈要**即时、柔和、可控**:擦到哪、哪里就从灰变亮;水声、水花、光芒都是正反馈。
- "干净"本身是奖励。整颗擦到 **100%** 无尘可"交付"，一声清脆奖励音收尾。

### 3. 核心循环
`覆尘水晶 → 喷水擦拭露出通透水晶 → 擦净 100% → 交付(奖励音，水晶入藏) → 播放随机换一枚新水晶重来`
交付过的水晶会被收进一个装满水的**成就宝瓶**;随时点右上角**画廊**按钮进去,看它们在水里漂浮、碰撞。凑够数量触发"全视之眼"完成仪式。

## 二、玩法设计原则

- **单指=擦拭,双指=看**:单指按住/拖动喷水冲刷;双指捏合缩放;拖空白背景旋转。职责清晰不打架。
- **擦拭是"揭示"而非"破坏"**:灰尘不是被"削掉"的实体，而是一层遮罩;擦拭把遮罩擦开，露出本就存在的光滑水晶。比"体素粉碎"更贴合"擦"的直觉，也更省。
- **随机与新鲜感**:▶️ 播放随机换一个水晶模型、随机的初始无尘擦痕、随机的入场旋转轴与角度(4 整圈 + 额外随机 0~180°，停在随机朝向)。
- **初始态要"像真擦过"**:开局的无尘区不是撒点，而是几笔**隐藏的随机擦拭**(真实笔刷沿表面走连贯路径)做到约 20% 无尘——看起来是"擦痕"而非"斑点"。
- **不保存半程进度，但保留成就收藏**:只持久化「日/夜」「挑战关」与**成就水晶队列**(交付过的水晶,先进先出)到 `user://`。**每次启动仍是一个新的随机未清洗水晶**——解压游戏重在"随手开一局"，而非续档;但你擦净交付的水晶会累积进成就宝瓶。
- **交付是仪式**:擦到 100% 后**屏幕下方中央**淡入圆圈(一声短"叮")，点击交付变对勾 + 奖励音，1 秒后对勾渐隐，转而在**右上角**淡入「画廊 + ▶️ 播放」双按钮。

### 底部/右上按钮的"活跃—闲置"逻辑
- 达标圆圈 / 交付对勾在**屏幕下方中央**(靠近底部)。
- 「画廊 + ▶️ 播放」双按钮在**右上角横向排列**，与左上日/夜按钮同大。
- **按状态分规则**(开局先显示,擦拭/旋转就淡出):**覆尘态**淡出后**只靠单击右上角区域**召回(不再静置自动回归——擦到一半界面清爽);**交付态**静置约 **1 秒**就自动淡回。让专注擦拭时界面干净，需要时快捷入口回来。
- 所有按钮按下都有"变大再缩回"的弹动反馈(绕中心 ~1.4×)。

### 成就宝瓶(画廊按钮进入)
右上角"画廊"按钮不再是选关轮播，而是打开一个**装满水的水晶宝瓶**——你交付过的每一颗水晶都收集在里面。

- **程序化圆瓶 + 水**:瓶身/水面用旋转体(lathe)脚本生成;瓶固定大小(不随数量变);水到瓶颈 50% 高度。水与玻璃都偏透，边缘略实。
- **CPU 物理**:每颗水晶有位置/速度/角速度——重力下沉、水阻力减速、**初始带随机自旋**;点击瓶身产生**一片冲击**把该柱水晶向上掀起并随机高速自旋、冒泡;水晶**两两碰撞**(均匀网格加速)。约束跟随瓶壁母线:可被冲进瓶颈,但**不越过水面**。
- **性能**:每种模型一个 `MultiMeshInstance3D`(一次 draw call 画任意数量);不透明水晶写深度靠硬件早 Z 天然遮挡剔除。
- **只绕竖直轴看**:拖拽绕 Y 旋转(带惯性)、固定 **30° 俯角**、缩放沿固定视线纯推拉不改变角度(瓶宽占屏 33%~80%);进场有一圈开场旋转(1 整圈 + 随机 0~180°，减速停)。灯光只留主光 + 环境光 + 神光。
- **左上灯光常驻**、右上"播放"= 退出并随机换关。
- **挑战与全视之眼**:成就水晶总数达标(30)→ 底部出现完成圆圈(可完成提示音)→ 点击 → 谜团音 + 圆圈中心变**全视之眼**(圆内线条眼)→ 瓶中爆发一团**巨大光芒 5 秒**、随机移除 15 颗、光散去 → 原位显示一行常驻文字「擦水晶 · 水晶磨き · Crystal Puzzle · v0.0.1」。瓶子**最多 30 颗**(先进先出:满了剔除最早那颗)。

## 三、视觉/手感/音效理念

- **纯黑背景 + 发光水晶**:突出主体，神光、反光在黑底上最亮眼。
- **灰尘哑光、水晶通透**:灰尘是纯漫反射的暗哑粉末;露出的水晶半透明、强反光、带蓝、能透看背面刻面。二者反差就是"擦"的爽点。
- **丁达尔/神光**:水晶的强反光高光通过屏幕空间体积光后处理(crepuscular rays)拉成光柱，在黑底空气中散射，像光透过水晶。
- **克制的粒子**:喷水水花是点缀不是主角——恒定透明度、松手整体淡出、数量与亮度都压在"看得见但不喧宾夺主"。
- **日/月两态**:☀️ 暖亮 ↔ 🌙 冷暗，一键切换整体光照与色温。
- **音效抽象**:换关/进关(动作)= 短促**按键音**;进入未完成/已完成场景与**交付**(结果)= **奖励庆祝音**;擦拭有持续水声。(早期的"呼声/迷团钟琴音"已废弃——事实证明简洁的按键音+庆祝音最不吵。)
- **启动画面**:把水晶从蓝底**抠图**(透明背景)放在纯黑加载页，像一颗悬浮发光的小水晶;App 图标则是带蓝底的完整方图(App Store 需不透明满图)。

## 四、技术架构与关键取舍

目标渲染后端是 **gl_compatibility**(Web + 老 iOS 都能跑)，很多"高级"特性不可用，于是大量用"够用且可控"的手法替代。

| 需求 | 方案 | 为什么 |
|---|---|---|
| 擦拭揭示，覆盖无上限 | **UV 遮罩贴图**(1024²，L8):擦拭画进遮罩，shader 按 `UV` 采样混合 | 最初用"最多 80 个冲刷点数组"会用尽、擦不干净;UV 遮罩无上限、边缘细腻，UV 分岛天然只作用命中那一面 |
| 灰尘要实、水晶要透 | **双趟材质(next_pass)**:灰尘层不透明写深度、剔背面;水晶层透明双面不写深度，各自按遮罩 `discard` 对方区域 | 单材质做不到"一面实一面透";两趟解决旋转穿模，又保留水晶通透可见刻面 |
| 折射丁达尔 | **屏幕空间体积光后处理**(CanvasLayer + `hint_screen_texture` 径向模糊)，而非真体积雾 | Compatibility 无体积雾;后处理在纯黑背景下效果好、可控 |
| 强反光(黑底无天空可反射) | shader 里**程序化"影棚"环境反光**(按反射方向生成亮斑)+ 边缘辉光 + 泛光 | 纯金属反光会反射成黑色;伪环境反光在黑底也能强反光 |
| 命中拾取擦拭点 | 射线命中 → 找**最近顶点的 UV** 画进遮罩;喷水粒子发射器先移到命中点再 restart | 物理射线拿不到 UV;最近顶点法足够。restart 前若不先移发射器，会在上次松手位置喷出一颗残留水珠 |
| 安全区 | 读 `DisplayServer.get_display_safe_area()`，按画布/屏幕缩放换算，放置左上/右上/底部中央按钮 | 避让刘海/灵动岛/圆角 |
| 完成判定 | 统计"遮罩已露出的顶点占比"，达 **100%** → 达标 | 近似表面无尘比例，稳定可控;初始擦拭也用它做到目标 20%。画廊未完成关也复用这套种子生成 |

### 光照
主平行光 + 前上方聚光 + 水晶中心一盏内部 OmniLight(透过露出处透内芒)+ 环境光;日夜切换统一调色温与能量。

## 五、工程踩坑与经验

1. **屏幕空间后处理会"看到"一切**。丁达尔神光读屏时会把**任何**足够亮的像素(包括叠加的水珠)拉成光柱/光球。→ 想排除某物，得让它**在后处理之后合成**。
2. **半透明 MIX 叠加会累积亮度**。很多 alpha 很低的水珠重叠，会累积成接近不透明的亮/暗团。→ 别让粒子在一点堆积(减阻尼飞散、限制数量、或不参与后处理)。
3. **`SubViewport.own_world_3d` 默认 false = 共享主世界**。把粒子丢进子视口"以为隔离了"，其实它还在主世界里带深度渲染。这正是那个"能被灰尘按深度遮挡、附着接触面、滞留 1 秒"的光球真凶。→ 要真正隔离，必须 `own_world_3d = true`。
4. **粒子回收有一帧空档**、松手残留:改成"擦时恒定、松手时整体材质透明度 tween 到 0"，并在再次擦拭时 `restart()` 前先把发射器移到当前触点。
5. **诊断法胜过猜测**:逐个"临时关掉某项看它还在不在"来二分，比反复猜快得多。
6. **按钮弹动/换关看不到动效**:▶️ 一按就 `_hide_bottom_ui()`，弹动来不及播。→ 换关延迟 ~0.22s 再执行，让 pop 播完。
7. **就地改写文件的致命顺序**:`open(f,"w").write(open(f).read().replace(...))` ——Python 先求值 `open(f,"w")`**立即把文件截断为 0**，再去 read 已空的文件，三个网页因此被清空。→ 就地替换必须先读进内存再写，或用 `sed -i`。
8. **JSON/GLB 打包**:`export_filter=all_resources` 不打包普通 `.json`，需在 `include_filter` 加 `*.json`;tripo 导出的 GLB 自带 UV，贴图或内嵌或外置，游戏用 `material_override` 覆盖材质渲染，故贴图不影响外观。

## 七、演进史(为什么它长这样)

一路重构过来:桌板令牌游戏(小魔典)→ 3D 拼图 → 宝石切割(体素粉碎)→ 覆粉钻石冲刷 → 定型为"擦水晶"。核心教训:**当机制和直觉/性能反复打架时，换一个更贴合直觉又更省的表达**(体素粉碎 → UV 遮罩揭示)，往往比死磕更好。
名字也一路变:小魔典 → 解谜小宝石 → 擦水晶;日文试过「クリスタル磨き」「宝石磨き」定为**「水晶磨き」**(擦・水・晶 三字日中同形);后转三语发行,英文名从 Crystal Polish 改为 **Crystal Puzzle**(商店名因须唯一用 Crystal Puzzle: Wipe & Polish),并加出 Android 包。
